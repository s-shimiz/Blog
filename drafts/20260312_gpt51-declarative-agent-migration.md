## はじめに

Microsoft 365 Copilot の拡張機能として利用できる **宣言型エージェント（Declarative Agent）** は、プロンプト（指示文）を記述するだけで業務に特化した AI エージェントを構築できる仕組みです。

2026 年に入り、Microsoft 365 Copilot の基盤モデルが **GPT 5.0 から GPT 5.1** にアップデートされました。これにより、エージェントの「指示の解釈の仕方」が大きく変わっています。

この記事では、公式ドキュメント「[GPT 5.1 以降のモデルの変更を理解する](https://learn.microsoft.com/ja-jp/microsoft-365-copilot/extensibility/declarative-model-migration-overview)」の内容を、初心者の方にもわかりやすいようにかみ砕いて解説します。

## そもそも宣言型エージェントとは？

宣言型エージェントとは、コードを書かずに **マークダウン形式の指示文（Instructions）** を書くだけで、Microsoft 365 Copilot 上にカスタムエージェントを作れる仕組みです。

たとえば「社内の経費精算ルールに基づいて質問に答えるエージェント」や「議事録を決まったフォーマットでまとめるエージェント」などを、プロンプトの記述だけで実現できます。

## GPT 5.0 と GPT 5.1 で何が変わったのか？

一言でいうと、**「指示を文字通りに実行するモデル」から「指示の意図を読み取って柔軟に実行するモデル」に変わった**ということです。

### GPT 5.0：文字通りに従う（リテラル優先）

| 特徴 | 説明 |
|---|---|
| 指示への忠実さ | 書かれた通りに正確に実行する |
| 書式・番号付け | 指定された通りに正確に適用する |
| 情報の補完 | 足りない情報があっても勝手に補わない |
| 予測可能性 | 高い（だが、指示が不完全だと精度が下がる） |

### GPT 5.1：意図を読み取って対応する（意図優先）

| 特徴 | 説明 |
|---|---|
| 指示の解釈 | 「何をしてほしいか」という意図を推測して動く |
| 推論の深さ | タスクの難易度に応じて自動的に調整する |
| 情報の補完 | 足りないステップを推測して埋める |
| 柔軟性 | 高い（だが、あいまいな指示に敏感になる） |

### 具体例で理解する

たとえば「メトリックを抽出して集計して」という指示を出した場合：

- **GPT 5.0** → 「抽出」と「集計」を 2 つの別々のステップとして順番に実行
- **GPT 5.1** → 2 つのステップをまとめて効率的に処理しようとする（意図しない結果になることも）

## GPT 5.1 で起きやすい問題

GPT 5.1 の「意図を読み取る」動作によって、以下のような予期しない変化が起きることがあります。

| 問題 | 具体例 |
|---|---|
| **手順の並べ替え** | 指定した順番通りに処理されない |
| **ステップの統合** | 「抽出して集計」のような複合指示が 1 つのステップにまとめられる |
| **トーンのドリフト** | 指定していないのに「教育的」や「おしゃべり」なトーンに変わる |
| **ステップの追加・削除** | モデルが推論して勝手にステップを追加または省略する |
| **番号付けの誤解釈** | 番号付きリストが「厳密な順序」として解釈され、意図しない順序で実行される |

## 新しい出力スタイル機能

GPT 5.1 では **8 種類の出力スタイル** が導入されました。指示文で明示的に指定できます。

| スタイル名 | 特徴 |
|---|---|
| **既定（Default）** | 詳しく説明的、先生のような口調 |
| **プロフェッショナル（Professional）** | 中立的で構造化されたビジネス口調 |
| **フレンドリー（Friendly）** | 会話的でサポーティブ |
| **率直（Straightforward）** | 直接的で簡潔 |
| **風変わり（Quirky）** | 表現力豊かでカジュアル |
| **効率的（Efficient）** | 最小限の情報量、結果重視 |
| **Nerdy** | 技術的で詳細志向 |
| **Cynical** | 懐疑的でドライ |

指示文にトーンを明記しない場合、GPT 5.1 が文脈から推測するため、意図しないスタイルになることがあります。**必ず明示的に指定しましょう。**

## 対応方法：2 つのアプローチ

GPT 5.1 への対応は、エージェントの用途に応じて **2 つのアプローチ** を使い分けます。

### アプローチ 1：厳密に制御する（固定プロセスが必要な場合）

業務ルールに厳格に従う必要がある場合（例：経費精算、法的文書の作成など）は、以下のポイントを押さえましょう。

#### ステップを明確に分割する

```markdown
## 処理手順

以下の手順を**順番通り**に実行してください。ステップをスキップまたは並べ替えないでください。

1. **ステップ 1**: ユーザーの質問からキーワードを抽出する
2. **ステップ 2**: ナレッジソースからキーワードに関連する情報を検索する
3. **ステップ 3**: 検索結果を箇条書きで要約する
4. **ステップ 4**: 要約が質問に正しく回答しているか確認する
```

#### 出力テンプレートを具体的に示す

```markdown
## 出力形式

以下のフォーマットで回答してください：

**回答:**
- 要点 1: （内容）
- 要点 2: （内容）
- 要点 3: （内容）

**参考情報:**
（出典を記載）
```

#### 重要な用語を定義する

```markdown
## 定義

- **売上高**: 当月の請求書発行額の合計（税抜き）
- **利益率**: （売上高 − 原価） ÷ 売上高 × 100

定義を発明しないでください。上記で指定されたもののみを使用してください。
```

#### GPT 5.0 互換のヘッダーを追加する（一時的な対策）

もともと GPT 5.0 向けに作ったエージェントをすぐに安定させたい場合は、指示文の先頭に「文字通りに実行してください」というガイダンスヘッダーを追加する方法も有効です。

### アプローチ 2：柔軟に任せる（ゴール重視の場合）

出力形式にこだわりがなく、目的の達成が重要な場合は、GPT 5.1 の柔軟な推論力を活かしましょう。

#### 目標だけを伝える

```markdown
## エージェントの目標

ユーザーからの問い合わせに対して、社内ナレッジベースを活用し、
わかりやすくフレンドリーなトーンで回答してください。
回答には必ず参考リンクを含めてください。
```

このように、細かいステップを指定する代わりに **「何を達成してほしいか」** を明確に伝えます。GPT 5.1 は目標から最適な手順を自分で組み立ててくれます。

#### ルーチン作業はモデルに任せる

GPT 5.1 は、明示的に指示しなくても「あいさつ」「お礼」などの丁寧な表現を自然に追加してくれます。こうした細部まで指定する必要はありません。全体のトーン（例：「フレンドリーで親切」）だけ設定すれば OK です。

## どちらのアプローチを選ぶ？判断フローチャート

```
エージェントの指示を見直したい
    │
    ├─ 固定の業務プロセスや書式に従う必要がある？
    │   ├─ YES → アプローチ 1（厳密に制御）
    │   │         ・ステップを明確に分割
    │   │         ・出力テンプレートを提示
    │   │         ・用語を定義
    │   │
    │   └─ NO → アプローチ 2（柔軟に任せる）
    │            ・ゴールを明確に伝える
    │            ・ガードレール（やってはいけないこと）を設定
    │            ・細かい手順は省略
    │
    └─ 更新後は必ずテスト！
        ・あいまいな質問を投げてみる
        ・エッジケースを試す
        ・トーンや書式が期待通りか確認
```

## まとめ

| 項目 | GPT 5.0 | GPT 5.1 |
|---|---|---|
| 指示の解釈 | 文字通り（リテラル優先） | 意図を推測（意図優先） |
| 推論の深さ | 固定的 | タスクに応じて動的に調整 |
| 出力スタイル | 一律 | 8 種類から選択可能 |
| 柔軟性 | 低い（予測しやすい） | 高い（あいまいさに敏感） |
| 最適な指示の書き方 | 細かくステップを記述 | 用途に応じて厳密/柔軟を使い分け |

**ポイントは 3 つです：**

1. **厳密さが必要な部分**は、ステップ・書式・用語を明確に定義する
2. **柔軟でよい部分**は、目標とガードレールだけ伝えてモデルに任せる
3. **更新後は必ずテスト**して、意図しない動作がないか確認する

GPT 5.1 への移行は、エージェントの指示文を見直す良い機会です。適切に対応すれば、より賢く柔軟なエージェントを実現できます。
