## はじめに

Microsoft 365 Copilot の **宣言型エージェント（Declarative Agent）** は、マークダウン形式の「指示文（Instructions）」を書くだけで、業務に特化した AI エージェントを構築できる仕組みです。

しかし、指示文の書き方次第で、エージェントの品質は大きく変わります。あいまいな指示を書けば、エージェントもあいまいな回答をしてしまいます。

この記事では、公式ドキュメント「[宣言型エージェントの効果的な命令を記述する](https://learn.microsoft.com/ja-jp/microsoft-365-copilot/extensibility/declarative-agent-instructions)」の内容を、初心者の方にもわかりやすく整理して解説します。

## 指示文の 3 つの基本コンポーネント

効果的な指示文には、以下の **3 つの基本コンポーネント** が必要です。

```
┌─────────────────────────────────────────┐
│              指示文の構成                 │
├─────────────────────────────────────────┤
│  1. 用途（Objective）                    │
│     → エージェントが何をするためのものか  │
│                                          │
│  2. ガイドライン（Guidelines）           │
│     → トーン、制限事項、一般的なルール    │
│                                          │
│  3. スキル（Skills / Workflow）           │
│     → 具体的なタスクの手順               │
└─────────────────────────────────────────┘
```

さらに、必要に応じて以下も追加します。

- 詳しい手順（ステップバイステップ）
- エラー処理と制限事項
- 対話の例（Few-shot プロンプト）
- 専門用語の定義
- フォローアップとクローズの方法

## ベストプラクティス 10 選

### 1. 明確で具体的な言葉を使う

エージェントへの指示は、**やるべきことを具体的に** 書きましょう。「適切に処理してください」のようなあいまいな表現は避けます。

| NG（あいまい） | OK（具体的） |
|---|---|
| データを処理してください | データから売上金額を抽出してください |
| 適切に対応してください | ユーザーの質問に箇条書きで回答してください |
| 情報を確認してください | `ServiceNow` でチケットのステータスを検索してください |

**ポイント**: 「ask」「search」「send」「check」「use」など、具体的な動詞を使いましょう。

### 2. ワークフローはステップバイステップで書く

複雑な処理は、1 つの大きな塊ではなく、**明確なステップに分割** しましょう。各ステップには「目標」「アクション」「次への移行条件」を含めます。

```markdown
## ステップ 1: 基本情報の収集
- **目標**: ユーザーの問題を特定する
- **アクション**: 不明確な場合は、1 つだけ質問をして明確にする
- **次へ**: 問題が明確になったらステップ 2 へ

## ステップ 2: 障害情報の確認
- **目標**: 既知の障害がないか確認する
- **アクション**: `ServiceNow` で現在の障害を検索する
- **次へ**: 障害がなければステップ 3 へ
```

### 3. タスクはアトミック（原子的）にする

1 つの指示に **複数のアクションを詰め込まない** でください。GPT 5.1 ではタスクが統合されて意図しない結果になることがあります。

| NG（複合） | OK（分割） |
|---|---|
| メトリックを抽出して結果を要約してください | **ステップ 1**: メトリックを抽出する<br>**ステップ 2**: 結果を要約する |

### 4. トーン・詳細度・出力形式は必ず指定する

指定しないと、モデルが勝手に推測して、バージョンごとに違う結果になります。

```markdown
## 応答ルール
- トーン: プロフェッショナルで簡潔
- 出力形式: セクションごとに 3 つ以内の箇条書き
- 要求された形式のみを返す（余計な説明は不要）
```

GPT 5.1 では 8 種類の出力スタイル（Professional、Friendly、Efficient など）が用意されています。詳細は[前回の記事](20260312_gpt51-declarative-agent-migration.md)をご覧ください。

### 5. マークダウンで構造化する

マークダウンの構造は、モデルが指示を理解するための**最も強力なシグナル**の 1 つです。

| 記法 | 用途 |
|---|---|
| `#`, `##`, `###` | セクション見出し |
| `-`（箇条書き） | 順序を問わない並列タスク |
| `1.`（番号付きリスト） | 順序が重要なステップ |
| `` ` ``（バックティック） | ツール名やシステム名（`Jira`、`ServiceNow` など） |
| `**`（太字） | 重要な指示の強調 |

**注意**: 順序が重要でないのに番号付きリストを使うと、モデルが「厳密な順番」と解釈してしまうことがあります。並列タスクには箇条書き（`-`）を使いましょう。

### 6. 専門用語は定義する

業界用語や社内用語は、モデルが誤って解釈する原因になります。必ず **定義セクション** を設けましょう。

```markdown
## 定義（この指示で使用する用語）
- **ARR**: Annual Recurring Revenue（年間経常収益）
- **チャーン率**: 解約率。計算式 = 解約顧客数 ÷ 期首顧客数 × 100
- **QBR**: Quarterly Business Review（四半期ビジネスレビュー）

定義を発明しないでください。上記で指定されたもののみを使用してください。
```

### 7. ツール・ナレッジソースは明示的に参照する

エージェントが使うべきツールやデータソースは、**各ステップで具体的に名前を挙げて** 指示しましょう。

```markdown
## ステップ 2: ナレッジベースの検索
- `ServiceNow KB` を使用して、ユーザーの問題に関連する記事を検索する
- SharePoint の社内ドキュメントも参照する
- コード インタープリターを使用してグラフを生成する
```

宣言型エージェントで参照できる主なソースは以下の通りです。

| ソースの種類 | 例 |
|---|---|
| アクション（API プラグイン） | `Jira` でチケットをフェッチ |
| Copilot コネクタ | `ServiceNow KB` でヘルプ記事を検索 |
| SharePoint / OneDrive | 社内ドキュメントを参照 |
| メール | ユーザーのメールで関連情報を確認 |
| Teams メッセージ | Teams チャット履歴を検索 |
| コード インタープリター | 棒グラフや円グラフを生成 |
| People ナレッジ | ユーザー情報をフェッチ |

### 8. 例を提示する（Few-shot プロンプト）

複雑なシナリオでは、**複数の例** を含めることでエージェントの理解が深まります。良い例と悪い例の両方を示すとさらに効果的です。

```markdown
## 対話の例

### 良い例
**ユーザー**: 「VPN に接続できません」
**エージェント**: 「具体的なエラーメッセージは表示されていますか？」
（ユーザー: 「DNS サーバーが応答しません」）
**エージェント**: 「障害情報を確認します...現在、障害は発生していません。
DNS 設定のリセットをお試しください。手順は以下の通りです...」

### 悪い例
- 「関連する記事を 15 件見つけました...」（情報過多）
- 「チケットを作成しました」（詳細を確認せずに実行）
```

### 9. 推論の深さをコントロールする

指示文の **言い回し** によって、モデルがどの程度深く考えるかをコントロールできます。

#### 深く考えてほしい場合

```markdown
深い推論を使用してください。問題をステップに分解し、各ステップを分析し、
代替案を評価し、最終的な判断を正当化してください。回答前に振り返りを行ってください。

タスク: 制約 A、B、C を考慮して、最適な 3 年間の移行戦略を決定する。
```

**キーワード**: 「分析」「評価」「正当化」「ステップバイステップで考える」「振り返る」

#### すぐに答えてほしい場合

```markdown
短い回答のみ。推論や説明は不要。最終結果のみを返してください。

タスク: この段落から製品名と更新日を抽出する。
```

**キーワード**: 「短い回答のみ」「説明不要」「最終結果のみ」

#### バランスを取りたい場合

```markdown
簡潔だが構造化された説明を提供してください。
短い要約、3 つの主要な要因、最終的な推奨事項を含めてください。
ステップバイステップの推論は不要です。
```

### 10. 自己評価ステップを追加する

指示文の最後に **セルフチェック** を入れることで、エージェントが回答する前に「指示通りにできているか」を確認するようになります。

```markdown
## 最終チェック（自己評価）
出力を確定する前に、以下を確認してください：
- すべての必要な項目が含まれているか
- 出力形式が指定通りか
- 推測や仮定に基づいた内容がないか
- 矛盾や不整合がないか
問題がある場合は、回答を修正してから返答してください。
```

## よくあるミスと対策

| よくあるミス | 何が起きるか | 対策 |
|---|---|---|
| **ツールの過剰呼び出し** | 必要な情報がないのにツールを呼び出す | 「必要な入力がある場合のみツールを呼び出す。ない場合はユーザーに確認する」と明記 |
| **言い回しのオウム返し** | 例の文言をそのまま使い回す | 複数の例を提示し、自然な言語での応答を促す |
| **過剰な説明** | 聞いていないことまで長々と説明する | 詳細度の制限と簡潔な出力例を追加する |

## 実践テンプレート集

公式ドキュメントで紹介されている **9 つのパターン** から、特に活用しやすいものを紹介します。

### パターン A: あいまいなタスクを確定的なワークフローに変換する

指示を「定義 → ステップ → 出力形式 → 自己評価」の 4 段構成にします。

```markdown
## タスク: メトリック計算と ROI 分析

### 定義（発明しないこと）
- 計算するメトリック: [メトリック 1], [メトリック 2], [メトリック 3]
- ROI の定義: ROI = (利益 - コスト) ÷ コスト
- データソース: 提供されたドキュメントのみ使用

### 手順（順番通りに実行。並べ替え禁止）
ステップ 1: ドキュメントから各メトリックの入力値を特定する。引用元のセクション名を記載する。
ステップ 2: 定義通りに各メトリックを計算する。入力が不足している場合は処理を中断し、不足項目をユーザーに質問する。
ステップ 3: 上記の定義に基づいて ROI を計算する。
ステップ 4: 以下の形式でのみ出力する。

### 出力形式
| メトリック | 値 | 出典 | 備考 |
|---|---|---|---|

### 最終チェック
すべてのメトリックに (a) 値、(b) 出典、(c) 仮定がないことを確認する。
仮定がある場合はユーザーに確認する。
```

### パターン B: 並列タスクと順次タスクを正しく分離する

並列と順次を混同すると、モデルが順番を間違えたりステップを統合したりします。

```markdown
## セクション A — データ抽出（並列・順不同）
- 価格変動を抽出する
- 利益率の変動を抽出する
- 顧客の感情テーマを抽出する

## セクション B — サマリー作成（順次・この順番で実行）
**ステップ 1**: セクション A のすべての結果を統合する
**ステップ 2**: 2 ページのコールプレップサマリーを作成する
```

### パターン C: 出力コントラクト（出力の約束事）

出力の形状・詳細度・トーンを厳密に定義するパターンです。

```markdown
## 出力コントラクト（必須）
- 目標: 四半期レビュー用のサマリーを作成する
- 形式: 箇条書き
- 詳細度: セクションごとに 3 項目以内
- トーン: プロフェッショナル
- 含める: 主要メトリック、リスク、次のアクション
- 含めない: 追加の推奨事項、補足説明、「役立つヒント」
```

### パターン D: リテラル実行ヘッダー（一時的な安定化）

GPT 5.0 から GPT 5.1 への移行で動作がおかしくなった場合の **応急処置** です。指示文の先頭に追加します。

```markdown
常に指示を文字通りに解釈してください。
意図を推測したり、不足しているステップを補完しないでください。
コンテキスト、推奨事項、仮定を追加しないでください。
ステップの順序を正確に守り、最適化しないでください。
要求された形式でのみ、簡潔に応答してください。
ステップで明示的に指示されない限り、ツールを呼び出さないでください。
```

**注意**: これは一時的な対策です。長期的には指示文全体を GPT 5.1 向けに最適化することをおすすめします。

## 指示文の開発は「反復プロセス」

指示文は一度書いたら終わりではありません。以下のサイクルを繰り返して改善していきます。

```
作成 → 発行 → テスト → 改善 → 作成 → ...
```

### テスト時のチェックポイント

1. **Microsoft 365 Copilot と比較**: エージェントが付加価値を出しているか
2. **会話スターターの動作確認**: 期待通りの回答が得られるか
3. **想定外の質問への対応**: ステップバイステップガイダンス以外の質問にも適切に対応できるか
4. **エッジケースのテスト**: あいまいな入力や例外パターンで問題がないか

## まとめ

| ベストプラクティス | ひとこと説明 |
|---|---|
| 明確で具体的な言葉を使う | 「処理する」ではなく「抽出する」「検索する」 |
| ステップバイステップで書く | 目標・アクション・遷移条件をセットで |
| タスクはアトミックにする | 1 指示 = 1 アクション |
| トーン・書式を明示する | 省略するとモデルが勝手に推測する |
| マークダウンで構造化する | 見出し・箇条書き・番号付きリストを使い分け |
| 専門用語は定義する | モデルの誤解を防ぐ |
| ツール・ソースは名前で指定 | 「検索する」ではなく「`ServiceNow KB` で検索する」 |
| 例を提示する | 良い例と悪い例の両方を示す |
| 推論の深さをコントロール | 言い回しで「深く考える/すぐ答える」を調整 |
| 自己評価ステップを追加 | 回答前にセルフチェックさせる |

**最も大切なこと**: 指示文の開発は一回で完成するものではありません。**作成 → テスト → 改善** のサイクルを回し続けることが、高品質なエージェントを作る近道です。
